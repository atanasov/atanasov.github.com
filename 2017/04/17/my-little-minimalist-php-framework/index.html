My little skeleton for smal php projects

Atanas Atanasov's Blog
2017-04-17

Let’s start with the fact that I’m not big fan of WordPress. At work I had to do many modifications on different WordPress projects. Trying different plugins, modifying themes to get closer to the features desired by the clients. Often customers change their minds and want totaly different functionality which were not foreseen. That means adding new plugins, changing plugins code, mixing 2 plugins into one. I don’t like the database structure and the whole components of WordPress. I tryied to like it but didn’t succeed. I researched a lot of PHP frameworks/CMS. I wanted something simple, easy to extend, lightweight. For Laravel I found AsgardCMS, which is really good and stable. It has modules approach so it can be extended easily. Every module has its own routes, models, controllers, repository, presenter and etc. I liked the idea, I tried using AsgardCMS for 1-2 projects. It was good in the beginning, but it was too heavy CMS because it’s based on Laravel. For small web apps hosted on shared hosting can be bad idea using Laravel. I am using Laravel, but it is sometimes tooo Fat for simple 3-4 page websites. The email sending was the only Laravel feature I could use and basic templating with Blade. The Idea So I decided to create simple skeleton for these small sites. I started it last year made few sites with it. It is still not ready to share it. Main features Lightweight, Simple, Module based, Easy Administration, Easy module creation Just add template, decide which content will be dynamic add is as variables in INI files, and set config settings. The Admin panel will create editable form, textbox, radio, checkboxes, select box, textarea automatically. Structure Many flat frameworks use, yaml for data structure. They add in single YAML file the content and the configuration of the page. Simple INI files are used to store the data, it’s not the perfect choice and there are many limitations. I cannot store complex arrays of content in INI file. But INI is simple, readable, no dependancy built-in in php parser and represents simple storage key->value mechanism. If the data structure is too compex the project is for more complex framework. Every page has its own INI data file. It can be used in admin to CRUD the data and in frontend to show the data. If I write the correct INI file admin panel will be automatically generated. Later I found some complex structures which could not fix in single page INI file. Also several dynamic data content were used on several pages. To solve this I added modules folder. Where many modules can be created with different structure and data and they can be used in any pages. They just have to be included in config INI file for the page. It’s like collection of objects or array with special behaviour like gallery, testimonials, feedbacks, books. The admin panel for modules is also autogenerated. There is no blog at this moment, but it can be created as module - all data for posts will be in single INI file or several. I use Flat file storage because of the portability of the whole project. Flat storage is not very reliable and at some level can be slow, but for small traffic websites its okay. I was thinking to use JSON or even SQLite and MySQL. But I will add PDO If I need it for example for blog and have good searching. For now the main purpose of this framework is creating simpe websites - most static content, rarely edited, easy for coding, easy for deploying and easy for maintaining. I will share it publicly when I finish it.
